# Federlet Shell and Remote Onboarding

Use this skill when helping add, modify, review, or troubleshoot a Federlet shell or remote application.

## 中文描述

这个 skill 用于协助 Federlet 项目中的 Shell 和 Remote 接入工作。适用场景包括：创建新 Shell、改造已有 Shell、创建或注册 Remote、检查 manifest 路由配置、校验 Shell/Remote 协议兼容性，以及确认 remote 暴露模块是否符合 `mount(context)` / `unmount()` 生命周期契约。

## First Read

Before changing files, inspect the relevant current code:

- `packages/cli/src/index.ts` for supported CLI flags.
- `packages/cli/src/commands/create-shell.ts` for generated shell templates.
- `packages/cli/src/commands/create-remote.ts` for generated remote templates.
- `packages/cli/src/utils/manifest.ts` for registration behavior.
- `packages/shared-types/src/index.ts` for `MicroAppContext`, `MicroAppInstance`, `RemoteRouteConfig`, and `RuntimeRemoteManifestItem`.
- `packages/shell-core/src/index.ts` plus `packages/react-shell` or `packages/vue-shell` for shell boundary behavior.
- Target shell config: `apps/shell-react/src/config/apollo.ts` or `apps/shell-vue/src/config/apollo.ts`.

## Choose the Track

**New shell?** Use "Shell Creation".

**Existing shell modification?** Use "Shell Modification".

**New or existing remote?** Use "Remote Onboarding".

When the task spans shell and remote, start with the shell runtime contract, then register the remote, then verify route loading end-to-end.

## Shell Creation

Prefer the existing CLI:

```bash
pnpm federlet create shell --framework react --name shell-example --port 3000
```

Supported shell frameworks are `react` and `vue`.

After generation, check that the shell has:

- Host configs for Rspack, Vite, Rsbuild, and Webpack.
- `@federlet/mf-runtime`, `@federlet/shared-types`, and the matching `@federlet/react-shell` or `@federlet/vue-shell` dependency.
- `src/config/apollo.ts` with `manifest.remotes`, `remoteSourcePolicy.allowedOrigins`, and `runtimeEnv`.
- `src/runtime-manifest.ts` using `bootstrapRuntimeRemoteRegistry`, `createRemoteDefinitionsFromManifest`, `registerRuntimeRemoteEntries`, and `SHELL_REMOTE_PROTOCOL_VERSION`.
- `src/remote-routes.ts` deriving fallback routes from the Apollo runtime config.
- `RemoteAppBoundary` wired to runtime routes and `runtimeRemoteRegistry`.

## Shell Modification

When adapting an existing shell to Federlet, preserve the host application's routing and layout, but add these integration points:

- Add a stable `SHELL_REMOTE_PROTOCOL_VERSION`; existing remotes must declare compatibility through `supportedShellProtocolVersions`.
- Load runtime environment from `window.__FEDERLET_ENV__` with a local Apollo fallback.
- Bootstrap remotes through `bootstrapRuntimeRemoteRegistry` instead of hand-maintaining a second route table.
- Render remotes with the React or Vue `RemoteAppBoundary`.
- Pass `runtimeRemoteRegistry` through `RemoteLoadOptions`.
- Keep `remoteSourcePolicy` aligned with remote `entryBaseUrl` origins.
- Keep source policy strict for production: prefer HTTPS, explicit origins, and no implicit localhost allowance.
- Keep remote routes data-driven; avoid duplicating manifest data in navigation, fallback routes, and route definitions.

## Remote Onboarding

1. Confirm the remote framework: `react`, `vue`, or `umi`.
2. Confirm the target shell: `react` or `vue`.
3. Confirm route details: `id`, `basename`, `path`, `title`, and `remoteName`.
4. Confirm remote entry details: `entryBaseUrl`, `remoteEntryType`, and `entryGlobalName` for `var` remotes.
5. Prefer the existing CLI before manual edits.
6. Verify manifest, protocol compatibility, and source policy after registration.

Create a remote with:

```bash
pnpm federlet create remote --framework react --name remote-example --route /example --port 3010 --title "Example Remote"
```

Register it in a shell Apollo mock with:

```bash
pnpm federlet register remote --shell react --id example --remote-name remote_example --basename /example --entry-base-url http://localhost:3010/ --title "Example Remote"
```

For Umi or other `var` remotes, include:

```bash
--remote-entry-type var --entry-global-name remote_umi_react
```

Do not hand-edit shell Apollo config unless the CLI cannot express the required change.

## Manifest Checks

Each shell manifest entry should satisfy:

- `id` is unique and stable.
- `remoteName` matches the Module Federation name generated by the remote config.
- `basename` starts with `/` and matches the route mounted inside the remote.
- `path` normally equals `<basename>/*`.
- `entryBaseUrl` is normalized with a trailing slash.
- `status` is `"active"` unless intentionally disabled.
- `supportedShellProtocolVersions` includes `SHELL_REMOTE_PROTOCOL_VERSION`.
- `remoteEntryType: "var"` has `entryGlobalName`.
- The origin of `entryBaseUrl` is allowed by `remoteSourcePolicy.allowedOrigins` when that shell config defines source policy.

For shell modification, also check that no separate hardcoded route list can drift from `DEFAULT_APOLLO_RUNTIME_CONFIG.manifest`.

## Mount Contract Checks

Every remote exposed module should export:

```ts
export function mount(context: MicroAppContext): MicroAppInstance | Promise<MicroAppInstance>
```

Check that the remote:

- Renders into `context.container`.
- Uses `context.basename` for router history or basename.
- Returns an object with `unmount()`.
- Cleans framework instances, subscriptions, timers, and event bus listeners in `unmount()`.
- Calls `context.onError?.(error)` for framework render errors when the framework supports it.
- Uses `context.eventBus` only through the injected context, not through direct cross-remote imports.
- Does not read sensitive Shell state from globals.

## Protocol Compatibility

Federlet compatibility is based on the Shell remote protocol version. When reviewing a remote registration:

1. Find the shell's `SHELL_REMOTE_PROTOCOL_VERSION`.
2. Ensure the remote manifest item includes it in `supportedShellProtocolVersions`.
3. If a remote is incompatible, prefer disabling it with `status: "disabled"` or updating its declared support rather than silently loading it.

When changing the shell protocol, treat it as a contract change: update generated shell templates, shell configs, tests, and any known remote compatibility declarations together.

## Recommended Verification

After creating or modifying a shell or remote, run focused checks first:

```bash
pnpm typecheck
pnpm test
pnpm build
```

For local smoke verification, run the target shell and remote together, then verify:

- Shell navigation shows the new route.
- The remote loads without source policy errors.
- Refreshing the remote route does not white-screen.
- Unmounting the route cleans event listeners and framework instances.
- Shell fallback routes and runtime manifest routes produce the same navigation shape.

## Output Format

When helping a user onboard or modify a shell/remote, summarize:

- The chosen shell and remote frameworks.
- The commands or files changed.
- The manifest entry and source policy impact.
- Any protocol, sandbox, or lifecycle risks.
- The verification commands run and their results.
